DBA Morning Health Check

5 checks. 5 minutes. Every morning.
Phase 5 — Work Like a Pro · Kudrant Lab

What this is: The exact Oracle SQL queries and commands for your daily morning health check. Run these five checks every morning before you do anything else. Save them in a script file so they're one copy-paste away.

Why it matters: A DBA who runs a morning checklist rarely gets blindsided. A DBA who skips it eventually gets a very bad surprise. This routine turns a 30-minute nervous check into a confident 5-minute habit.

How to use it: Run each query in order. Read what it tells you. Act on anything that looks wrong. Document and move on.

1

Check Instance Status & Uptime

⏱ ~30 seconds
Why: Confirm every database is up, open, and has been running since you expect. If an instance restarted overnight without your knowledge, you need to know immediately.
SQL — Run as SYS or DBA-privileged user
-- Check instance status and uptime SELECT instance_name, status, database_status, startup_time, ROUND((SYSDATE - startup_time), 2) AS uptime_days FROM v$instance; -- Check database open mode SELECT name, open_mode, log_mode, database_role FROM v$database;
What a healthy result looks like

STATUS = OPEN, DATABASE_STATUS = ACTIVE, OPEN_MODE = READ WRITE. Uptime should match when you expect the database was last started. If uptime is suspiciously low (e.g., 0.2 days when it should be 30+), the instance restarted — investigate why.

✅ If healthy
  • Confirm and move to Check 2
🚨 If not healthy
  • Instance restarted? Check alert log immediately for the cause
  • Status not OPEN? Investigate before users notice
2

Verify Last Night's Backups

⏱ ~45 seconds
Why: A backup that silently failed is exactly the kind of thing that ends careers. Check completion status, not just whether it started. Know when your last successful backup was — that's your real exposure window.
SQL — RMAN backup history
-- Last 7 days of RMAN backup status SELECT session_key, input_type, status, TO_CHAR(start_time, 'YYYY-MM-DD HH24:MI') AS start_time, TO_CHAR(end_time, 'YYYY-MM-DD HH24:MI') AS end_time, ROUND(elapsed_seconds/60, 1) AS duration_min FROM v$rman_backup_job_details WHERE start_time > SYSDATE - 7 ORDER BY start_time DESC; -- Quick check: when was the LAST successful backup? SELECT MAX(end_time) AS last_successful_backup FROM v$rman_backup_job_details WHERE status = 'COMPLETED' AND input_type IN ('DB FULL', 'DB INCR');
What a healthy result looks like

Last night's backup shows STATUS = COMPLETED. The last successful backup should be within the last 24 hours. If STATUS = FAILED or COMPLETED WITH WARNINGS, or if the last successful backup is more than 24 hours old, act immediately.

✅ If healthy
  • Backup completed — you're protected. Move on.
🚨 If failed
  • Check the RMAN log for the failure reason
  • Fix the cause and run a manual backup NOW
  • Never let a failed backup slide — fix same morning
3

Scan the Alert Log for Errors

⏱ ~60 seconds
Why: The alert log is the database's diary. If something went wrong overnight — an ORA error, a space warning, a failed process — it's recorded here. Catching it now means you act before users report symptoms.
SQL — Query alert log (12c and later)
-- Last 24 hours of alert log errors and warnings SELECT TO_CHAR(originating_timestamp, 'YYYY-MM-DD HH24:MI:SS') AS alert_time, message_type, message_text FROM v$diag_alert_ext WHERE originating_timestamp > SYSDATE - 1 AND message_type IN (1, 2) -- 1=ERROR, 2=WARNING ORDER BY originating_timestamp DESC;
Alternative — Linux command line (any version)
# Tail the last 100 lines of the alert log tail -100 $ORACLE_BASE/diag/rdbms/<db_name>/<instance>/trace/alert_<instance>.log # Search for ORA- errors in the last 24 hours grep -i "ORA-" $ORACLE_BASE/diag/rdbms/<db_name>/<instance>/trace/alert_<instance>.log | tail -20
What a healthy result looks like

No rows returned, or only informational messages (not errors). Common benign entries include scheduled job completions and log switches. ORA- errors, especially ORA-00600 (internal error), ORA-07445 (OS exception), or space-related errors need investigation.

✅ If clean
  • No errors overnight — great. Move on.
🚨 If errors found
  • Read the full error in context — what happened before and after?
  • Search the ORA code to understand the cause
  • Assess severity: is it impacting users right now?
4

Check Tablespace & Archive Space

⏱ ~45 seconds
Why: Space issues are the most preventable emergencies. A tablespace that hits 100% stops the database from writing. An archive destination that fills up can hang the entire instance. Catch it at 85% and fix it calmly.
SQL — All tablespace usage in one query
-- Tablespace usage summary (shows % used) SELECT df.tablespace_name, ROUND(df.total_mb, 1) AS total_mb, ROUND(df.total_mb - NVL(fs.free_mb, 0), 1) AS used_mb, ROUND(NVL(fs.free_mb, 0), 1) AS free_mb, ROUND(((df.total_mb - NVL(fs.free_mb, 0)) / df.total_mb) * 100, 1) AS pct_used FROM (SELECT tablespace_name, SUM(bytes)/1024/1024 total_mb FROM dba_data_files GROUP BY tablespace_name) df LEFT JOIN (SELECT tablespace_name, SUM(bytes)/1024/1024 free_mb FROM dba_free_space GROUP BY tablespace_name) fs ON df.tablespace_name = fs.tablespace_name ORDER BY pct_used DESC;
SQL — Archive log destination space
-- Check archive destination usage SELECT dest_name, status, destination, space_used, space_limit, ROUND((space_used/NULLIF(space_limit,0))*100, 1) AS pct_used FROM v$archive_dest_status WHERE status = 'VALID';
Linux — Check disk space
# Overall disk space on the server df -h # Focus on Oracle-related mount points df -h | grep -iE "oracle|data|arch|redo|backup"
What a healthy result looks like

All tablespaces below 85% used. Archive destination has ample free space. Disk mounts are not approaching capacity. Anything over 85% is a yellow flag. Anything over 95% is a red flag — act immediately.

✅ If under 85%
  • All clear — move to Check 5
🚨 If over 85%
  • 85-95%: plan to add space today — add datafiles or extend
  • 95%+: act NOW — this can crash the database within hours
  • Archive full: purge old logs or add space immediately
5

List Failed Scheduler Jobs

⏱ ~30 seconds
Why: Scheduled jobs run critical operations overnight — stats gathering, data loads, cleanup tasks. A failed job might not alert you directly, but its downstream effects (slow queries, space buildup, stale data) will become someone's urgent complaint if you don't catch it first.
SQL — Failed scheduler jobs in the last 24 hours
-- Failed DBMS_SCHEDULER jobs in the last 24 hours SELECT job_name, status, TO_CHAR(actual_start_date, 'YYYY-MM-DD HH24:MI') AS start_time, error#, SUBSTR(additional_info, 1, 200) AS error_info FROM dba_scheduler_job_run_details WHERE actual_start_date > SYSDATE - 1 AND status = 'FAILED' ORDER BY actual_start_date DESC; -- Also check for long-running jobs still executing SELECT job_name, state, TO_CHAR(last_start_date, 'YYYY-MM-DD HH24:MI') AS started, ROUND((SYSDATE - last_start_date) * 24, 1) AS running_hours FROM dba_scheduler_jobs WHERE state = 'RUNNING';
What a healthy result looks like

No failed jobs returned. If the second query returns running jobs, check whether the run time is normal for that job. A stats-gathering job running for 2 hours may be fine; a 5-minute cleanup job running for 8 hours is not.

✅ If no failures
  • All jobs completed — morning check is done
🚨 If jobs failed
  • Read the error info to understand why
  • Determine impact: is this job critical or can it wait?
  • Rerun the job manually if needed, or fix the underlying cause

Your 5-Minute Morning Routine — Summary

  1. Instance status & uptime — confirm every database is OPEN and running since you expect (~30 sec)
  2. Backup verification — confirm last night's backup COMPLETED, know your last successful backup (~45 sec)
  3. Alert log scan — check for ORA errors or warnings overnight (~60 sec)
  4. Space check — tablespaces, archive, and disk all under 85% (~45 sec)
  5. Failed jobs — no failed scheduler jobs, no abnormally long-running jobs (~30 sec)
Total: ~3.5 minutes once you have the queries saved. Add coffee time and you're at 5.
Pro tip: Save all five queries in a single SQL file on your desktop. Every morning, open it, run them in order, read the results. Within a week this becomes muscle memory. Within a month you won't even think about it — you'll just do it, and you'll never be blindsided again.
